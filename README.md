## 🛠 기술 스택 구성 및 활용 영역

본 프로젝트는 **민감 정보 기반 간편 세금 계산 서비스**라는 특성을 고려하여,  
각 기술 스택을 **역할과 책임 기준으로 분리**하여 사용하였습니다.

---

### 🟢 Runtime & Language

![Node.js](https://img.shields.io/badge/Node.js-18.x-339933?style=for-the-badge&logo=node.js&logoColor=white)
![JavaScript](https://img.shields.io/badge/JavaScript-ES2022-F7DF1E?style=for-the-badge&logo=javascript&logoColor=black)

#### Node.js (18.x)
- 서버리스 환경(AWS Lambda)에서 실행되는 **세금 계산 API 런타임**
- 비동기 처리 기반으로 요청 단위 계산에 적합
- 계산 후 즉시 종료되는 구조로 데이터 잔존 위험 최소화

**사용 영역**
- 부가가치세 / 소득세 / 근로소득세 계산 API
- 서버리스 함수 실행 환경

#### JavaScript (ES2022)
- 최신 문법을 활용해 계산 로직을 **명확한 함수 단위**로 분리
- 복잡한 세무 규칙을 단순화한 계산 모델 구현

**사용 영역**
- 세금 계산 로직
- 입력값 검증 및 결과 포맷팅

---

### ☁️ Cloud & Infrastructure

![AWS](https://img.shields.io/badge/AWS-Serverless-232F3E?style=for-the-badge&logo=amazonaws&logoColor=white)

#### AWS Serverless Architecture
- 민감 정보를 저장하지 않기 위해 **서버리스 기반 아키텍처 채택**
- 요청 단위 실행 후 종료되는 구조로 보안성과 확장성 확보

**사용 영역**
- **AWS Lambda**: 세금 계산 로직 실행
- **API Gateway**: HTTPS 요청 수신 및 라우팅
- **IAM**: 서비스 간 접근 권한 최소화

---

### 🔐 Security-Oriented Design

![Security](https://img.shields.io/badge/Security-Data%20Minimization-critical?style=for-the-badge)
![Architecture](https://img.shields.io/badge/Architecture-Stateless-blue?style=for-the-badge)

#### Data Minimization (데이터 최소 수집)
- 사용자 입력 금액을 **DB에 저장하지 않음**
- 모든 계산은 **In-memory 방식**으로 처리 후 즉시 폐기

**적용 영역**
- 세금 계산 요청 전 과정
- 데이터 저장 계층 미사용 설계

#### Stateless Architecture
- 서버가 상태를 가지지 않는 구조
- 요청 간 데이터 공유 없음

**적용 영역**
- 모든 계산 API
- 서버리스 함수 설계 전반

---

### 📊 Logging & Monitoring (제한적 활용)

![Security](https://img.shields.io/badge/Logging-Minimal-lightgrey?style=for-the-badge)

#### Minimal Logging Policy
- 로그에는 **금액 정보 미포함**
- 서비스 안정성을 위한 최소한의 메타 정보만 기록

**기록 항목**
- 요청 시각
- 계산 유형 (부가세 / 소득세 등)
- 처리 성공 여부
- 응답 시간

클라우드 환경에서 민감 데이터를 다루는 서비스의 설계 방향을 보여주는 데 목적이 있습니다.

## 📌 프로젝트 개요

**TaxMate**는 개인사업자 및 근로자의 세금 부담을  
**간단하게 추정해볼 수 있도록 돕는 계산 서비스**입니다.

초기 구현 단계에서는 세금 계산 로직의 정확성보다  
**사용자 경험(UX)**과 접근성을 우선하여,  
복잡한 세무 규칙을 단순화한 계산 모델을 사용하였습니다.

그러나 서비스가 실제 금액(매출, 소득, 급여 등)을 입력받는 구조인 만큼,  
고도화 단계에서는 계산 정확도보다  
**민감 정보 보호와 클라우드 보안 설계**가 핵심 과제가 됩니다.

이에 따라 본 프로젝트의 클라우드 고도화 방향은  
**“정확한 신고 시스템”이 아닌,  
“민감 정보를 남기지 않는 안전한 계산기”**를 목표로 설정하였습니다.

## 🧠 주요 기능

모든 결과는  
**“납부해야 할 금액”이 아닌  
“약 ○○원 수준의 예상 금액”**으로 제공됩니다.

## ⚠️ 계산 방식 안내 (중요)

본 서비스의 계산 로직은 **참고용 시뮬레이션 모델**입니다.

- 실제 국세청 신고 기준과 차이가 있을 수 있습니다.
- 인적공제, 세액공제, 각종 예외 규정은 일부 생략되어 있습니다.
- 부가가치세는 업종별 평균 부가가치율을 적용한 추정치입니다.

```text
본 서비스의 계산 결과는 참고용이며,
정확한 세무 신고는 세무사 또는 국세청 기준을 따르시기 바랍니다.
```

#### 장점

- 디스크에 데이터가 남지 않음
- 인스턴스 장기 유지로 인한 공격 표면 감소
- 자동 확장 및 트래픽 대응 가능

이를 통해 **보안성과 운영 효율을 동시에 확보**합니다.

### 3. 인증·권한 분리 중심의 IAM 설계

클라우드 고도화 단계에서는 기능보다 **권한 분리**가 중요합니다.

각 컴포넌트의 역할은 다음과 같이 분리됩니다.

- **API Gateway**: 요청 검증 및 전달
- **계산 서비스**: 세금 계산 로직만 수행
- **로그 시스템**: 요청 성공 여부 등 메타 정보만 기록

어떠한 컴포넌트도 입력된 금액 전체를  
장기적으로 보유하거나 조회할 수 없도록 설계하여,  
내부 오남용 및 사고 발생 시 피해 범위를 최소화합니다.

---

### 4. 로그 및 모니터링의 제한적 활용

운영 안정성을 위해 로그와 모니터링은 활용하되,  
**금액 정보는 로그에 포함하지 않습니다.**

로그에 포함되는 정보는 다음과 같습니다.

- 요청 시각
- 계산 유형 (부가세 / 소득세 등)
- 처리 성공 여부
- 응답 시간

이를 통해 서비스 품질은 유지하면서도  
민감 정보 노출을 방지합니다.

---

### 5. 계산 결과 표현 방식의 보수적 설계

본 서비스는 **법적 효력을 가지는 세무 계산기가 아님**을 명확히 합니다.

- “납부해야 할 세금” ❌  
- **“예상 세금”** ⭕

- 정확한 금액 ❌  
- **“약 ○○원 수준”** ⭕

이를 통해 사용자 오해를 방지하고,  
서비스의 역할과 책임 범위를 명확히 정의합니다.

---

### 6. 단계적 고도화 전략

본 프로젝트는 다음과 같은 **단계적 확장 전략**을 가집니다.

- **1단계**: 입력 데이터 비저장 구조의 계산 API
- **2단계**: 사용자 동의 하 요약 결과 선택적 저장
- **3단계**: 인증, 암호화, 감사 로그를 포함한 실서비스 수준 확장

각 단계는 **보안 수준과 책임 범위를 명확히 구분**하여 진행합니다.
